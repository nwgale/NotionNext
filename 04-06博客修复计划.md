# 04-06 博客修复计划

## 目标定义
- 目标：恢复 `https://tianfei.chat` 正常展示文章内容，避免再次发布空站。
- 完成标准：首页与文章页恢复正常，GitHub Actions 连续多次执行无空数据发布，出现异常时自动中止并提示检修。

## 协作方式
- 代码修改在本地仓库进行。
- 每个可测试步骤完成后立即验证并记录结论。
- 每次发布到 GitHub 后，观察 Actions 日志与线上结果，再决定是否迭代。

## 执行步骤（按可测试粒度）

### 编号1：建立诊断基线（只加日志，不改业务行为）
- 范围文件：`lib/notion/getAllPageIds.js`、`lib/db/getSiteData.js`
- 动作：补充关键诊断日志，打印 `collectionId`、`viewIds`、`collection_query` 命中情况、空列表来源。
- 可测试结果：在 Actions 日志中能清晰看到“空列表由哪一层返回”。
- 风险控制：仅加日志，不改变线上行为。

### 编号2：加发布闸门（先止血）
- 范围文件：`.github/workflows/deploy-to-domestic.yml`
- 动作：健康检查增加“`posts.length === 0` 或关键字段缺失则失败退出”。
- 可测试结果：当构建结果为空时，工作流在 rsync 前终止，服务器不被空内容覆盖。
- 风险控制：只影响异常构建，不影响正常发布。

### 编号3：修复数据判定逻辑（防空数据误判为成功）
- 范围文件：`lib/db/getSiteData.js`
- 动作：将“空 `allPages`”定义为不可用数据；在 `EXPORT/CI` 场景直接抛错中止。
- 可测试结果：当 Notion 返回异常结构时，构建失败并给出明确错误。
- 风险控制：保留现有 fallback 机制，避免误伤正常数据。

### 编号4：增强 `getAllPageIds` 容错（减少偶发空结果）
- 范围文件：`lib/notion/getAllPageIds.js`
- 动作：在 `collectionQuery[collectionId]` 缺失时，采用更稳妥的 fallback 逻辑并输出诊断信息。
- 可测试结果：同样输入条件下，`pageIds` 获取成功率提升，空列表概率下降。
- 风险控制：保留 try/catch，避免直接抛出 TypeError。

### 编号5：发布并观察一次完整链路
- 动作：提交到 `main`，触发 `Deploy to Domestic Server`，检查三处：
  - Actions 构建日志
  - 健康检查结果
  - `https://tianfei.chat` 首页与随机文章页
- 可测试结果：线上恢复，且日志无“空列表被发布”。

### 编号6：连续观察与迭代
- 动作：观察至少两个周期（含定时触发），确认稳定性。
- 可测试结果：连续运行均不出现空站发布；若出现异常，能自动 fail 并定位原因。

## 回滚与应急策略
- 若修复后出现误拦截：
  - 先回滚健康检查阈值配置。
  - 保留诊断日志，继续定位真实触发条件。
- 若线上再次出现空站：
  - 立即停止自动部署触发。
  - 使用上一版稳定产物重新同步服务器。

## 验收清单
- 首页可见真实文章列表，不是空白站。
- 随机 3 篇文章页可正常访问。
- Actions 异常时能在 rsync 前中止。
- 日志能明确定位失败层级（API / collection_query / pageIds / 过滤逻辑）。
